1 API Gateway vs Load Balancer

Q: At a high level, what distinguishes an API Gateway from a Load Balancer? In a microservices architecture, where does each sit, and why would you use both?

A Load Balancer operates at transport/network layer (L4) or application layer (L7), distributing incoming traffic across multiple backend servers for availability and fault tolerance. It’s typically transparent, using algorithms like round‑robin or least connections.

An API Gateway works exclusively at L7, acting as a reverse proxy that routes, composes, and transforms API requests. It centralizes cross‑cutting concerns: authentication, rate limiting, request/response transformation, logging, and routing based on path/header. It may also aggregate responses from multiple microservices.

Using both: In a .NET deployment, a load balancer (e.g., Azure Load Balancer or AWS ALB) distributes traffic among instances of the API Gateway for high availability. The gateway then routes to individual microservices, applying policies. This decouples scaling of the gateway tier from backend services.

Real‑world .NET example: YARP (Yet Another Reverse Proxy) can be configured as an API Gateway in ASP.NET Core. Behind it, internal services communicate via gRPC or HTTP, while an L4 load balancer ensures multiple gateway instances are available.

2 Reverse Proxy vs Forward Proxy

Q: Explain the fundamental difference between a forward proxy and a reverse proxy. Give a scenario where each is used in a .NET ecosystem.
  • Forward Proxy: Sits between internal clients and the internet. Clients explicitly configure it; the proxy makes requests on their behalf (e.g., hiding client IPs, enforcing corporate web filtering). In .NET, you might set `HttpClientHandler.Proxy` to route outbound calls through an enterprise proxy.
  • Reverse Proxy: Sits in front of web servers, intercepting requests from the internet. It provides load balancing, SSL termination, caching, and security. The client thinks it’s talking to the origin server directly. In ASP.NET Core, YARP or IIS ARR serves as a reverse proxy.

Real‑world scenario: A .NET back‑end might use a forward proxy (e.g., Squid) to scrape external APIs while masking IPs. A reverse proxy (like YARP) shields internal microservices, terminates TLS, and routes by paths.

3 Horizontal scaling vs Vertical scaling

Q: Compare horizontal and vertical scaling. As a Lead Engineer, how do you decide which approach is appropriate for a .NET monolith vs a set of microservices?
  • Vertical scaling (scaling up): Adding more resources (CPU, RAM) to a single server. Simple but hits physical limits, creates a single point of failure, often requires downtime.
  • Horizontal scaling (scaling out): Adding more instances. Demands statelessness or distributed state management. In .NET, session state must be externalized (e.g., Redis via `IDistributedCache`) and file storage moved to blob storage or a network share.

Decision factors: Monoliths often initially scaled vertically because they are stateful. Beyond a point, horizontal scaling requires containerization and careful session management. Microservices are designed for horizontal scaling; each service can scale independently. Kubernetes or Azure App Service auto‑scale adjusts instance counts.

Vertical scaling is cheaper until you need extreme scale; horizontal scaling increases operational complexity (service discovery, distributed tracing, eventual consistency). A .NET lead plans for horizontal scaling from the start if traffic is unpredictable.

4 Microservices vs Monolithic architecture

Q: When would you argue against breaking an existing well‑structured .NET monolith into microservices?

Microservices add substantial complexity. Reasons to stay monolithic:

  • Team size & maturity: Small teams can maintain a modular monolith with clear interfaces, avoiding distributed‑system overhead.
  • Performance requirements: Monoliths have zero network latency between modules; breaking them adds serialization, network hops, and eventual consistency challenges.
  • Data integrity: Strong ACID transactions across domains are trivial in a monolith but require sagas or 2PC in microservices.
  • Development velocity: A monolith allows simpler debugging, integrated testing, and deployments until the codebase becomes unmanageable.
  • Cost: Running dozens of microservices with separate databases, message brokers, and orchestrators is expensive for low‑traffic apps.

Many successful projects start as an ASP.NET Core modular monolith using MediatR and vertical slices, only extracting services when a module requires independent scaling or a different tech stack.

5 Caching strategies

Q: Walk through the major caching patterns (cache‑aside, read‑through, write‑through, write‑behind). Which one would you use for a high‑read product catalog in an ASP.NET Core API, and why?
  • Cache‑aside (lazy loading): Application first checks cache; on miss, fetches from DB and populates cache. Control logic lives in application code.
  • Read‑through: Cache sits between app and DB; application only talks to cache. Cache loads data on miss transparently.
  • Write‑through: Writes go to cache, which synchronously writes to DB. Cache always up‑to‑date but increases write latency.
  • Write‑behind (write‑back): Writes go to cache, which asynchronously flushes to DB. Lower write latency, risk of data loss.

For a product catalog (read‑heavy, infrequent writes): Cache‑aside is simple and widely supported. Using `IDistributedCache` (Redis) with `Microsoft.Extensions.Caching.StackExchangeRedis`, the code checks Redis; on miss, fetches from SQL Server and caches with TTL. Change propagation via cache invalidation through pub/sub (e.g., Redis Pub/Sub or a message queue) when a product updates.

Lead‑level: set appropriate TTLs, use staggered expiration to avoid thundering herds, and consider stampede prevention (e.g., `GetOrCreateAsync` with a semaphore or lock).

6 Rate limiting

Q: Design a rate‑limiting strategy for a multi‑tenant SaaS API built with ASP.NET Core where each tenant has different subscription levels.

Requirement: Per‑tenant limits (Free: 100 req/min, Pro: 1000 req/min) and global DDoS protection.

Algorithm: Token bucket (allows bursts), sliding window, or fixed window. Token bucket is often preferred for smoothing.

Implementation in .NET: Use the built‑in `System.Threading.RateLimiting` namespace (introduced in .NET 7/8) with `AddRateLimiter()` middleware. A custom `IRateLimiterPolicy` resolves the tenant from JWT claim/API key and selects the appropriate `RateLimiterOptions`. Store state in a distributed cache (Redis) for multi‑instance consistency. For extreme scale, use Redis‑Lua scripts for atomic counters.

Architecture: Place rate‑limiting at the API Gateway (YARP) to offload backend services, and propagate headers like `X-RateLimit-Remaining`.

7 Single Sign‑On (SSO)

Q: You need to implement SSO across three ASP.NET Core web apps on different subdomains. What authentication protocol would you choose, and how would you configure the middleware?

Protocol: OpenID Connect (OIDC) with OAuth 2.0 is the modern standard. SAML may be used for legacy enterprise providers, but OIDC is simpler and widely supported in .NET.

Implementation:

  • Run an identity provider (Duende IdentityServer, Azure AD) at `auth.example.com`.
  • Each app uses `AddOpenIdConnect()` with authorization code flow + PKCE for SPAs if needed.
  • Share authentication cookie across subdomains by setting `Cookie.Domain = ".example.com"`.
  • Configure the OIDC options with `Authority`, `ClientId`, `ClientSecret`, `ResponseType = "code"`, and `SaveTokens = true`.
  • For API backends, the same authority issues JWT access tokens; APIs validate them with `AddJwtBearer()`.

Security: Enforce HTTPS, use `SameSite=Lax`/`Strict` cookies, configure CORS, and use token refresh with sliding expiration for long‑lived sessions.

8 Apache Kafka

Q: What makes Kafka extremely fast for high‑throughput messaging, and how would you integrate it into a .NET event‑driven system using Confluent.Kafka?

Speed factors:

  • Sequential disk I/O: Append‑only log segments, leveraging sequential writes.
  • Zero‑copy transfer: `sendfile()` syscall from page cache to socket buffer without copying to user space.
  • Batch processing & compression: Producers batch messages; compression (LZ4, Zstd) reduces network/storage overhead.
  • Partitioning: Topics are split into partitions for parallelism across brokers and consumers.
  • Page cache reliance: Frequently accessed data sits in OS page cache, avoiding disk reads.

Integration in .NET: Use `Confluent.Kafka` with `ProducerBuilder` and `ConsumerBuilder`. For reliability, use idempotent producers (`EnableIdempotence = true`), manual offset commits, and implement the outbox pattern with EF Core to atomically update DB and publish events. For high availability, configure multiple bootstrap servers and retries.

10 JWT vs OAuth vs SAML

Q: Many developers confuse JWT, OAuth, and SAML. Clarify the roles each plays, and describe a scenario where you would use all three together in a .NET ecosystem.
  • JWT (JSON Web Token): Compact, URL‑safe token format carrying claims. Often used as access tokens, ID tokens, or refresh tokens.
  • OAuth 2.0: Authorization framework allowing third‑party apps to obtain limited access. Defines flows (authorization code, client credentials) but not token format.
  • SAML (Security Assertion Markup Language): XML‑based standard for exchanging authentication/authorization data, mainly for enterprise SSO.

Scenario: A corporate .NET app uses SAML to authenticate against Active Directory Federation Services (AD FS). The identity server (Duende IdentityServer) accepts the SAML assertion and issues a JWT access token (per OAuth 2.0) for the API gateway and downstream services. SAML for legacy IDP, OAuth 2.0 for modern token‑based security, JWT as the lightweight token.

.NET libraries: `Sustainsys.Saml2`, `AddOpenIdConnect()`, `AddJwtBearer()`.

11 Database scaling

Q: You are architecting a .NET e‑commerce platform that initially runs on a single SQL Server instance but must scale to handle 10x read traffic and 5x write traffic. Outline a comprehensive scaling strategy.

Read scaling:

  • Add read replicas (Always On Availability Groups, Azure SQL Geo‑Replication).
  • Implement read‑write split in data access layer, routing queries to replicas via custom `DbContext` or libraries like `EFCoreSecondLevelCacheInterceptor`.
  • Introduce Redis cache for frequently accessed data (product details, pricing).
  • Materialized views or denormalized read models via CQRS.

Write scaling:

  • Vertical scaling first; then sharding by tenant/region.
  • Message queue (RabbitMQ/Kafka) to buffer writes, eventual consistency for non‑critical operations.
  • Optimize connection pooling (`Max Pool Size`).

Supporting patterns: Outbox pattern + change data capture (Debezium) for syncing. Consider NoSQL (Cosmos DB, DynamoDB) for specific workloads if relational limits are exceeded.

Follow‑up: "How do you implement sharding transparently in Entity Framework Core, and what are the limitations?"

12 Database replication vs sharding

Q: Differentiate database replication from database sharding. When would you choose one over the other, and how does the .NET application layer change?
  • Replication: Copies data to one or more replicas. Provides read scalability, HA, DR. Writes go to primary, reads can be routed to replicas. .NET needs read/write splitting (connection strings or routing `DbContext`).
  • Sharding: Splits data into independent partitions based on a shard key. Enables write scalability because writes are distributed. Application must route to the correct shard, often via a shard map manager or custom logic based on tenant ID.

Choice: Replication when read load overwhelms a single instance but write load is manageable. Sharding when dataset size or write throughput exceeds a single server’s capacity, or for multi‑tenant isolation.

Application changes: replication requires minimal code changes; sharding demands careful key design, cross‑shard query constraints, and potential use of sagas for data spanning shards.

13 JWT vs session‑based authentication

Q: As a Lead Engineer, you must choose between JWT‑based and session‑based authentication for a new ASP.NET Core web app that serves both a server‑rendered UI and a public API. Compare the two and explain your decision.
  • JWT: Stateless, client stores token. Server validates signature, no server‑side session storage. Ideal for APIs, SPAs, mobile. Challenges: revocation, token size. .NET: `AddJwtBearer()`.
  • Session‑based: Server creates session, stores cookie with session ID. Stateful; requires session storage (Redis). Simple revocation, smaller client footprint. .NET: `AddCookie()` with `ISession` or `ITicketStore`.

Hybrid approach: Register both schemes. Cookie auth for browser interactions (MVC/Razor), JWT for API calls via `Authorization: Bearer` header, using schemes policies (`[Authorize(AuthenticationSchemes = "...")]`). This gives the best of both worlds. Public API gets JWT; server‑rendered part benefits from cookie auth’s ease of invalidation and built‑in CSRF protection.

14 Consistent hashing

Q: Explain consistent hashing and its advantages over simple modulo‑based distribution. How would you implement a consistent hash ring in a .NET distributed caching layer using Redis?

Consistent hashing maps nodes and keys to a circular space (0 to 2^32‑1). Adding/removing a node remaps only a fraction of keys, minimizing cache invalidation. Modulo `hash(key) % N` invalidates almost all keys when N changes.

In .NET: Use a `SortedDictionary<long, string>` to represent the ring. Compute hash (e.g., MurmurHash) for each node with virtual nodes to balance load. For a key, find the closest node in the ring via binary search. In practice, distributed cache clients like `StackExchange.Redis` with Redis Cluster handle consistent hashing internally; you rarely write it from scratch. Understanding it is essential when designing sharded services or custom key‑value stores.

15 Message queues

Q: Describe how message queues improve system resilience and scalability. Walk through a specific scenario in .NET where you would use an asynchronous command queue with dead‑letter handling.

Benefits: decoupling, buffering, resilience (retries, dead‑letter), scalability (competing consumers).

Scenario: Order placement API receives high load. Instead of synchronous inventory/payment/email, it publishes `PlaceOrderCommand` to an Azure Service Bus queue. A .NET `BackgroundService` with `ServiceBusProcessor` consumes messages. On transient failure, retries 3 times with exponential backoff, then moves to dead‑letter queue. Another process monitors DLQ and alerts operations.

Implementation can use `Azure.Messaging.ServiceBus` or MassTransit with RabbitMQ, which provides retry, circuit breaker, and DLQ middleware.

16 REST vs gRPC

Q: When would you choose gRPC over REST for service‑to‑service communication in a .NET microservices ecosystem? What performance and tooling trade‑offs exist?

gRPC advantages: HTTP/2 multiplexing, binary Protobuf serialization → lower latency, smaller payloads, strong typing, bidirectional streaming, built‑in deadlines/cancellation. In .NET, `Grpc.AspNetCore` provides first‑class support.

REST advantages: Human‑readable JSON, wide tooling (curl, Postman, Swagger), browser‑friendly.

For internal .NET‑to‑.NET microservices where latency matters, gRPC is strongly preferred. For external/public APIs, REST with JSON remains standard due to interoperability. gRPC’s binary nature complicates debugging; you may need `grpcurl`.

Lead note: In a polyglot environment, ensure all platforms have solid gRPC support. You can expose dual REST/gRPC APIs using ASP.NET Core’s ability to run both, but weigh the maintenance cost.

17 Leader election (Raft, Paxos)

Q: Why is leader election necessary in distributed systems, and how do consensus algorithms like Raft and Paxos achieve it? Provide an example of how you’d use leader election in a .NET distributed background job scheduler.

Leader election ensures a single node is responsible for a task, avoiding duplicate execution and split‑brain. Raft achieves this via randomized timers and majority voting; Paxos is harder to implement but also provides consensus.

In .NET: Don’t write Raft from scratch. Use external coordination services:

  • Kubernetes lease mechanism (using `Microsoft.KubernetesClient`).
  • Azure Blob leases – acquire a lease on a blob; only one instance can hold it.
  • Apache ZooKeeper / etcd with .NET clients.

Example: A distributed cron job runner where only one pod should execute monthly billing. Using Kubernetes leases, each pod competes; the leader runs the job, others stand by. If the leader fails, its lease expires and another instance takes over.

18 Circuit breakers

Q: Describe the circuit breaker pattern and how you’d implement resilient HTTP calls between .NET microservices using Polly.

Circuit breaker prevents cascading failures by detecting repeated faults and temporarily stopping calls to a failing service.

  • Closed: normal operation; failures increment counter.
  • Open: after threshold, circuit opens and calls immediately fail (fast-fail).
  • Half‑Open: after timeout, limited trial calls allowed; success returns to Closed, failure re‑opens.

In .NET with Polly: Combine with `IHttpClientFactory`.

builder.Services.AddHttpClient("orders")
    .AddTransientHttpErrorPolicy(p => p.WaitAndRetryAsync(3, retryAttempt => 
        TimeSpan.FromSeconds(Math.Pow(2, retryAttempt))))
    .AddPolicyHandler(Policy.Handle<HttpRequestException>()
        .AdvancedCircuitBreakerAsync(0.5, TimeSpan.FromSeconds(30), 10, TimeSpan.FromSeconds(60)));

The circuit breaker opens after 50% failures over 30s with min 10 actions, then stays open 60s. Often combined with a fallback policy for degraded responses.

19 Failover and disaster recovery

Q: Design a failover and disaster recovery plan for a .NET‑based e‑commerce platform hosted on Azure. Define RTO, RPO, and the architectural choices.

RPO (Recovery Point Objective): Max acceptable data loss, e.g., < 1 minute (near real‑time replication).

RTO (Recovery Time Objective): Max downtime, e.g., < 15 minutes.

Architecture:

  • Active‑Active across two Azure regions using Azure Front Door for global load balancing and failover.
  • Data: Azure SQL with active geo‑replication, auto‑failover groups. Cosmos DB multi‑region writes with conflict resolution. Blob storage GRS.
  • Services: AKS or App Services deployed via IaC (Bicep/Terraform) to both regions.
  • Messaging: Azure Service Bus geo‑disaster recovery.
  • Failover process: Front Door health probes shift traffic; database failover group triggers forced failover if needed. Runbooks automate the process.

Regular DR drills, chaos engineering, and ensuring the .NET app handles transient faults (Polly) and reconnects gracefully after failover.

Follow‑up: "How do you test and validate your disaster recovery plan for a .NET system without impacting production users?"

20 Heartbeats and health checks

Q: How do you implement comprehensive health checks for an ASP.NET Core microservice, and how are they consumed by Kubernetes for self‑healing?

ASP.NET Core provides `Microsoft.Extensions.Diagnostics.HealthChecks`.

builder.Services.AddHealthChecks()
    .AddCheck("self", () => HealthCheckResult.Healthy())
    .AddDbContextCheck<AppDbContext>("sql")
    .AddRedis("redis", "connection_string")
    .AddUrlGroup(new Uri("https://external.api/health"), "external_api");

app.MapHealthChecks("/health/live", new HealthCheckOptions { Predicate = _ => false }); // liveness: only self
app.MapHealthChecks("/health/ready", new HealthCheckOptions { Predicate = _ => true }); // readiness: all

Kubernetes `livenessProbe` and `readinessProbe` point to these endpoints. Kubelet restarts unresponsive containers (liveness) and removes unready pods from service (readiness). This enables zero‑downtime deployments and self‑healing. Health checks should be non‑intrusive and may incorporate circuit‑breaker logic.

21 CQRS

Q: Explain CQRS and its advantages. In a .NET application, when would you avoid it, even if you anticipate high read/write asymmetry?

CQRS separates read and write models. Commands update the write model (normalized, relational) and publish events that eventually update the read model (denormalized, optimized for queries). Benefits: independent scaling, simpler queries, technology flexibility.

When to avoid:

  • Simple domains where a single EF model suffices; CQRS adds unnecessary complexity.
  • Tight consistency requirements: The read model is eventually consistent, which can cause UI surprises. May require compensating patterns.
  • Team expertise: Event handling, eventual consistency, and out‑of‑order delivery increase cognitive load.
  • Not true asymmetry: If reads and writes are similar, a well‑cached CRUD model may be enough.

In .NET, using MediatR for command/query separation is not full CQRS; it’s just in‑process separation. Start simple; introduce CQRS only when proven scale or query complexity demands it.

22 Event‑driven architecture

Q: Design an event‑driven order processing system using .NET and Apache Kafka for an e‑commerce platform. What patterns ensure reliable publishing and consumption?

Events: `OrderPlaced`, `PaymentProcessed`, `InventoryReserved`, `OrderFulfilled`.

Reliability patterns:

  • Transactional outbox: Order service inserts order row and event record in the same EF Core transaction; a separate publisher reads the outbox and publishes to Kafka (at‑least‑once).
  • Idempotency: Consumers store consumed event IDs; process only once, using database unique constraints.
  • Kafka’s idempotent producer (`EnableIdempotence = true`) and manual offset commits after processing.
  • Schema evolution via Confluent Schema Registry/Avro for backward compatibility.
  • Retry with backoff, dead‑letter topics, circuit breakers (Polly).

Tooling: MassTransit with Kafka transport simplifies many of these concerns.

23 Microservices vs Monolith – Deep Comparison

Q: Beyond surface‑level differences, what are the deep organizational, DevOps, and technical trade‑offs between microservices and a well‑designed modular monolith in .NET?
  • Organizational coupling: Microservices align with cross‑functional teams (Conway’s Law). A modular monolith can achieve similar decoupling with clear interfaces, but enforcement is harder.
  • Deployment complexity: Microservices require container orchestration, service discovery, distributed tracing, per‑service CI/CD. A monolith has a single pipeline, simpler debugging, but can slow down as it grows.
  • Data consistency: Monoliths use ACID transactions; microservices force eventual consistency, sagas, compensating transactions, which are harder to test.
  • Scalability: Microservices allow independent scaling of hotspots; monolith scales as a whole, wasting resources.
  • Technology diversity: Microservices allow different databases/languages; but for a .NET shop, a monolith with one stack often suffices.
  • Testing: E2E testing in microservices is challenging; contract testing becomes crucial. Monolith E2E tests are easier but slower.

Even with a large team, a modular monolith may be preferable if the domain is not well‑understood, or if strong transactional integrity is needed. Premature microservices lead to costly refactoring. Use architecture tests (NetArchTest) to enforce module boundaries.

Follow‑up: "How would you enforce module boundaries in a .NET modular monolith to prevent tight coupling, using tools like NetArchTest or Roslyn analyzers?"

24 Saga pattern

Q: Describe the Saga pattern for managing distributed transactions across microservices. Compare orchestration‑based and choreography‑based sagas with a .NET example.

A saga is a sequence of local transactions, each updating a single service and publishing an event or command. If a step fails, compensating transactions undo preceding steps.

  • Choreography: Services listen to events and react. Loose coupling, but flow becomes hard to track. Example: `BookingService` emits `FlightBooked`; `HotelService` reacts; on failure emits `HotelBookingFailed`, and `BookingService` cancels the flight.
  • Orchestration: A central saga orchestrator tells each service what to do. Tracks state, sends compensating commands on failure. Easier to understand, but can become a single point of coupling.

In .NET, orchestration can use state machine libraries like **Stateless** or **Workflow Core**, persisting saga state. MassTransit implements sagas with its state machine and Courier routing slip. Use the outbox pattern to atomically publish commands/events with local DB changes.

Example: travel booking saga — `CreateBooking` → Reserve Car → Reserve Hotel → Process Payment. If payment fails, cancel hotel and car.

25 Idempotency

Q: Why is idempotency critical in distributed systems, and how would you enforce it in a .NET REST API that processes payments?

Importance: Network failures, client retries, or message redelivery can cause duplicate requests. Without idempotency, a payment might be processed twice, leading to double charges.

Implementation in .NET API:

  • Client sends a unique `Idempotency-Key` header.
  • Server checks a store (Redis or DB table) for the key. If not found, process, store response with key and TTL (e.g., 24h). If found, return stored response without reprocessing.
  • Redis: atomic `SETNX` with TTL. SQL: unique constraint on `IdempotencyKey` column; pre‑insert a placeholder with status `Processing`, then update to `Completed` after payment.
  • Custom middleware or `IAsyncActionFilter` can handle this cross‑cutting concern. Libraries like `IdempotentAPI` exist.

Lead considerations: propagate idempotency keys across downstream calls; design all operations to be idempotent (e.g., use PUT with ETags). Combine with saga pattern for multi‑service idempotency.
