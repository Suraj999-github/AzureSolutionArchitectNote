Understanding Authentication
in .NET Core

A clear, beginner-friendly guide to securing your modern applications

🔐 Why Authentication Matters

Authentication is the process of verifying who a user (or service) is before granting access to your application. In modern systems, it's not optional — it's essential. Here's why:

  • Protects sensitive data — Prevents unauthorized access to personal information, financial records, and business data.
  • Enforces access control — Ensures that only the right people can perform specific actions (e.g., admin vs. regular user).
  • Builds user trust — Users expect their accounts to be safe; breaches destroy reputation.
  • Compliance & regulations — GDPR, HIPAA, PCI-DSS, and other frameworks mandate proper authentication.
  • Prevents common attacks — Mitigates risks like credential stuffing, session hijacking, and impersonation.

📊 "Which authentication method should I pick?"

📚 Authentication Methods Breakdown

Below are the most common authentication approaches in .NET Core. Each serves a different purpose — understanding them will help you pick the right one.

🔑 JWT Bearer Authentication Best for APIs

JSON Web Tokens (JWT) are compact, self-contained tokens that carry user claims (like name, email, roles). The server signs them with a secret or private key, so they can be verified without hitting a database on every request — making them stateless and perfect for scalable APIs.

  • How it works: Client sends the token in the Authorization: Bearer <token> header. The server validates the signature and extracts claims.
  • Built-in support: AddAuthentication().AddJwtBearer() in ASP.NET Core.
  • Typical use: REST APIs, microservices, SPAs (React/Angular/Vue), mobile backends.
  • Pros: Stateless, scalable, cross-platform, no server-side session storage needed.
  • Cons: Tokens can't be easily revoked before expiry; token size can grow with many claims.

🔐 OAuth 2.0 Authorization Framework

OAuth 2.0 is an authorization framework — not strictly authentication — that allows a user to grant limited access to their resources without sharing credentials. It's the backbone behind "Log in with Google/Facebook/GitHub" flows.

  • Key grants (flows): Authorization Code (most secure, for web apps), Client Credentials (M2M), Device Code (TVs/IoT), PKCE (for SPAs/mobile).
  • Typical use: Delegated access, third-party integrations, "Login with X" buttons.
  • Note: OAuth 2.0 alone doesn't tell you who the user is — that's where OIDC comes in.

🆔 OpenID Connect (OIDC) SSO Standard

OpenID Connect is a thin identity layer built on top of OAuth 2.0. It adds an ID Token (a JWT) that contains verified user identity information, enabling true Single Sign-On (SSO).

  • ID Token vs Access Token: ID Token = who the user is; Access Token = what they can access.
  • Typical use: SSO across multiple apps, enterprise identity (Azure AD, Okta, Auth0, IdentityServer).
  • .NET Core support: AddOpenIdConnect() for web apps; AddJwtBearer() can validate OIDC-issued tokens for APIs.

🍪 Cookie Authentication Traditional Web Apps

The classic approach: after a successful login, the server creates an encrypted cookie containing the user's session/identity. The browser automatically sends it with every subsequent request.

  • How it works: AddAuthentication().AddCookie() — the server manages the cookie, encrypts it, and validates it on each request.
  • Typical use: Server-rendered apps (MVC, Razor Pages), where the frontend and backend are tightly coupled.
  • Pros: Simple to implement, automatic cookie handling by browsers, easy to invalidate (clear the cookie).
  • Cons: Not ideal for SPAs or mobile apps; vulnerable to CSRF if not properly protected; requires server-side session state or cookie encryption.

👥 ASP.NET Identity User Management System

ASP.NET Core Identity is a full-featured membership system. It handles user registration, password hashing, email confirmation, two-factor authentication (2FA), role management, and external logins — all out of the box.

  • What it provides: User store (with EF Core), password policies, lockout, 2FA (TOTP), social login integration, and customizable UI (Scaffolded Identity pages).
  • Pairs perfectly with: Cookie Auth (for web apps) or JWT (for APIs — Identity can issue JWTs).
  • Typical use: Any app that needs a full user database with registration, login, and profile management.

🪟 Windows Authentication Intranet / Enterprise

Uses the underlying Windows OS to authenticate users via NTLM or Kerberos. The user's Windows login is seamlessly passed to the web app — no separate login form needed.

  • Configuration: AddAuthentication().AddNegotiate() or AddNtlm(); enable Windows Authentication in IIS/Kestrel.
  • Typical use: Internal corporate apps, intranet portals, behind a VPN, Active Directory environments.
  • Pros: Zero login friction for domain users; leverages existing AD infrastructure.
  • Cons: Windows-only; not suitable for internet-facing apps or non-Windows clients.

🗝️ API Key Authentication Simple M2M

A straightforward method where clients include a pre-shared secret key in the request (usually in a custom header like X-API-Key). The server checks it against a known list.

  • How it works: Often implemented via custom middleware or an AuthenticationHandler that reads the header and validates the key.
  • Typical use: Simple service-to-service calls, webhooks, public APIs with registered consumers, prototyping.
  • Pros: Extremely simple to implement and understand; easy to revoke individual keys.
  • Cons: Not suitable for user-facing auth; keys can leak; no built-in expiry or claims mechanism; less secure than token-based approaches.

📜 Certificate Authentication (mTLS) High-Security M2M

Mutual TLS (mTLS) requires both the client and server to present X.509 certificates. The server validates the client certificate against a trusted Certificate Authority (CA), ensuring strong, hardware-backed identity.

  • Configuration: AddAuthentication().AddCertificate() in ASP.NET Core; requires proper certificate setup on both ends.
  • Typical use: Zero-trust architectures, financial services, government systems, service mesh (e.g., Istio), highly regulated environments.
  • Pros: Very high security; no passwords or tokens to leak; mutual trust established at the transport layer.
  • Cons: Complex certificate management (issuance, rotation, revocation); higher operational overhead.

🤔 When to Use Each Method

  • Building a REST API consumed by a SPA or mobile app? → Use JWT Bearer tokens. Stateless, scalable, and works everywhere.
  • Need "Log in with Google/Microsoft/Facebook"? → Use OAuth 2.0 + OpenID Connect. OIDC gives you verified identity; OAuth handles delegated access.
  • Building a traditional server-rendered web app (MVC/Razor)? → Use Cookie Authentication + ASP.NET Identity. It's simple, battle-tested, and integrates seamlessly.
  • Internal corporate app on a Windows domain? → Use Windows Authentication. Users get seamless SSO with their AD credentials.
  • Two backend services talking to each other? → Use API Key (for simplicity) or mTLS Certificate (for high security). Also consider OAuth 2.0 Client Credentials flow.
  • Need a full user database with registration, 2FA, roles? → Use ASP.NET Identity as your user store, then pair it with Cookie Auth (web) or JWT (API).
  • Zero-trust, highly regulated environment? → Use mTLS Certificates for mutual, hardware-backed trust.

💡 Key Takeaways

🧠 The Golden Rule of Authentication

  • Authentication choice depends entirely on your application's needs. There is no single "best" method — only the best fit for your scenario.
  • APIs ≠ Web Apps. APIs need stateless tokens (JWT); web apps thrive with session-based cookies.
  • OAuth 2.0 is for authorization; OIDC is for authentication. Together, they power modern SSO.
  • ASP.NET Identity is not an auth method — it's a user store. It works alongside Cookie Auth or JWT to manage users, passwords, and roles.
  • Security is a spectrum. API Keys are simple but less secure; mTLS is complex but highly secure. Choose based on your threat model.
  • Combine approaches when needed. A common pattern: OIDC for login → issues a JWT → JWT used for API calls. This gives you SSO + stateless API auth.

✅ Practical Recommendations for Modern Systems

🚀 Start Here — Proven Patterns

  • 🌐 For Web APIs (SPA backends, mobile backends, microservices):
    Use JWT Bearer tokens. Combine with OAuth 2.0 + OIDC if you need SSO. Example stack: ASP.NET Core + AddJwtBearer() + IdentityServer / Azure AD / Auth0 as the token issuer.
  • 🖥️ For Server-Rendered Web Apps (MVC, Razor Pages, Blazor Server):
    Use ASP.NET Identity + Cookie Authentication. It's the most straightforward, well-documented path. Add OIDC for external logins (Google, Microsoft, etc.).
  • 🏢 For Enterprise SSO (multiple apps, one login):
    Use OpenID Connect + OAuth 2.0 with a central identity provider (Azure AD, Okta, Duende IdentityServer). This gives users one login across all your apps.
  • 🤖 For Service-to-Service (microservices talking to each other):
    Use OAuth 2.0 Client Credentials flow (modern) or mTLS Certificates (high-security). Avoid API Keys for production inter-service auth if possible — they're harder to rotate securely.
  • 🛡️ Always: Use HTTPS (TLS) everywhere, hash passwords (Identity does this automatically), implement rate limiting, and plan for token/key rotation from day one.
