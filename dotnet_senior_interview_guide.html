01 Your ASP.NET application is experiencing high response times under load. How would you identify bottlenecks and optimize performance? Performance
// Diagnosis Strategy
1
Establish observability baselines — instrument with Application Insights or OpenTelemetry to capture request duration, CPU, memory, and thread-pool saturation before any changes.
2
Run load tests (k6, NBomber, or Azure Load Testing) to reproduce the bottleneck in a staging environment at realistic concurrency levels.
3
Capture a memory dump or CPU profile under load using dotnet-trace, PerfView, or Visual Studio Diagnostic Tools to identify hot paths.
4
Query slow-query logs and Application Insights dependencies to see if latency is server-side or DB/network.
// Common Bottlenecks & Fixes
  • Synchronous blocking: Audit all I/O paths. Replace blocking .Result / .Wait() calls with async/await. Thread-pool starvation is the #1 ASP.NET Core perf killer.
  • Over-fetching from DB: Add projection (Select), pagination, and check for missing indexes. Use AsNoTracking() for read-only EF Core queries.
  • Missing response caching: Apply IMemoryCache for hot data, distributed Redis cache for multi-instance, and HTTP response caching headers for CDN.
  • Excessive allocations: Use Span<T>, ArrayPool, and RecyclableMemoryStream for high-throughput paths to reduce GC pressure.
  • Middleware overhead: Profile middleware pipeline; remove or short-circuit unused middleware for critical paths.
// Targeted Code Pattern
// ❌ Blocking — starves thread pool under load
var result = _repository.GetUserAsync(id).Result;

// ✅ Async all the way down
var result = await _repository.GetUserAsync(id);

// ✅ Read-only EF Core — skip change tracker
var users = await _ctx.Users
    .AsNoTracking()
    .Where(u => u.IsActive)
    .Select(u => new UserDto(u.Id, u.Name))
    .ToListAsync(cancellationToken);
Senior Signal
Emphasise that you measure before and after each change using benchmarks (BenchmarkDotNet) or APM dashboards — not intuition. Premature optimisation without data is a code smell in itself.
02 You inherit a legacy monolithic .NET application with frequent issues. How would you refactor it into a scalable architecture? Architecture
// Guiding Principle: Strangler Fig Pattern

Never attempt a full rewrite. Incrementally carve out bounded contexts into independently deployable services while the monolith continues serving traffic.

// Phased Approach
1
Understand & map the domain — run Event Storming sessions with domain experts. Identify bounded contexts, seams, and high-churn areas.
2
Improve observability first — add structured logging (Serilog), distributed tracing, and health checks. You can't safely change what you can't observe.
3
Decouple data — separate DB schemas per bounded context; break FK dependencies. Use integration events (via MassTransit + RabbitMQ/Azure Service Bus) to sync state.
4
Introduce an API Gateway (YARP or Ocelot) to route traffic. Extract one bounded context at a time as a .NET Minimal API or gRPC service.
5
Containerise (Docker) and deploy on Kubernetes or Azure Container Apps. Establish a service mesh (Dapr) for cross-service concerns.
// Architecture Checklist
Within Each Service
  • Clean / Vertical Slice Architecture
  • CQRS with MediatR
  • Own database schema (schema-per-service)
  • Repository + Unit of Work
Cross-Cutting
  • Async messaging (Service Bus / RabbitMQ)
  • Outbox pattern for reliable events
  • Centralised identity (Duende IdentityServer)
  • Centralised logging & tracing (Seq / Grafana)
Senior Signal
Mention the organisational dimension: Conway's Law means team topology must mirror the target architecture. Propose aligning teams around bounded contexts, not technical layers.
03 A production issue is causing memory leaks in your application. How would you diagnose and fix it? Reliability
// Triage Without Recycling

First, confirm the leak: is working set growing monotonically? Check w/ dotnet-counters monitor for gc-heap-size, gen-2-gc-count, and loh-size. Do not blindly restart — you'll lose the evidence.

// Diagnostic Toolkit
  • dotnet-dump / createdump — capture a process dump under high memory, then analyse offline with dotnet-dump analyze using dumpheap -stat to find which types dominate the heap.
  • PerfView / Visual Studio Memory Snapshot — compare two heap snapshots at intervals to see which object counts are growing.
  • Application Insights Live Metrics + Memory Alerts — set up proactive alerts on private bytes for early warning.
// Root Causes & Fixes
  • Event handlers not unsubscribed: The publisher holds a reference to the subscriber. Always unsubscribe in Dispose(), or use WeakReference / weak events.
  • Static collections as caches: Unbounded static dictionaries grow forever. Replace with IMemoryCache with size limits and expiration.
  • IDisposable not disposed: Use using or register services with the correct DI lifetime. HttpClient instances created with new are a classic leak.
  • Captured closures in long-lived objects: Lambdas capturing large object graphs in background services or timers prevent GC.
  • LOH fragmentation: Allocating many byte arrays >85 KB. Pool them with ArrayPool<byte>.Shared.
// Safe Cache Pattern
// ❌ Unbounded static dictionary — classic leak
private static Dictionary<string, HeavyObject> _cache = new();

// ✅ Bounded IMemoryCache with eviction
_cache.Set(key, value, new MemoryCacheEntryOptions
{
    SlidingExpiration = TimeSpan.FromMinutes(5),
    Size = 1
});
Senior Signal
Walk through your actual post-mortem process: dump → dumpheap → identify root type → trace GC roots → find the retaining path. Showing you know how to find the retaining chain demonstrates experience most candidates lack.
04 Your application needs to scale globally with high availability. How would you design the architecture? Scalability
// Target Architecture
  • Multi-region active-active deployment on Azure (or AWS) with Azure Front Door / Traffic Manager for global load balancing and geo-routing.
  • Stateless application layer — all state externalised to distributed cache (Azure Cache for Redis) and database, so any instance can handle any request.
  • Horizontal pod autoscaling on AKS (KEDA for event-driven) — scale to zero for non-critical services, scale out in seconds for traffic spikes.
  • Cosmos DB or Azure SQL Hyperscale — globally distributed, multi-master writes, automatic failover with guaranteed RPO < 5 s / RTO < 30 s.
  • CDN (Azure CDN / Akamai) — serve static assets and cache API responses at edge for latency < 20 ms globally.
  • Async messaging (Service Bus / Event Hub) to decouple write paths from read paths; use Saga pattern for distributed transactions.
// Resilience Patterns
Availability
  • Circuit Breaker (Polly v8 / Resilience Pipeline)
  • Retry with exponential back-off + jitter
  • Health checks + readiness probes
  • Chaos engineering (Azure Chaos Studio)
Observability
  • OpenTelemetry → Azure Monitor
  • Distributed tracing across services
  • SLO dashboards (99.9% / 99.95%)
  • Error budgets & alerting runbooks
Senior Signal
Define your RTO and RPO targets up front and show how each architectural decision maps to those SLAs. Architects who can reason from business requirements to infrastructure choices stand out.
05 A new feature introduces database performance issues. How would you optimize queries and data access? Data Access
// Immediate Diagnosis
1
Enable EF Core query logging (or MiniProfiler) in staging to capture generated SQL. Look for N+1 queries, table scans, and missing WHERE predicates.
2
Run EXPLAIN ANALYZE (Postgres) or Execution Plan (SQL Server) on slow queries. Identify seq scans, nested loop joins on large tables, and key lookups.
3
Check for lock contention via sys.dm_exec_requests or pg_locks — sometimes it's not the query, it's blocking.
// Optimisation Toolbox
  • Fix N+1: Use .Include() / .ThenInclude() in EF Core or explicit JOINs in Dapper. Load aggregates, not individual rows in a loop.
  • Composite indexes: Index on the exact (column_a, column_b) pair used in WHERE + ORDER BY. Covering indexes eliminate key lookups.
  • Projections: Never SELECT *. Reduce I/O by selecting only needed columns; map to DTOs in the query layer.
  • Read replicas / CQRS read side: Route read-heavy queries to a replica or a dedicated read model (materialised view / Elasticsearch) to offload the primary.
  • Compiled queries: Use EF Core EF.CompileAsyncQuery() for hot paths to avoid repeated query compilation overhead.
  • Batching writes: Use ExecuteUpdateAsync / ExecuteDeleteAsync (EF Core 7+) or bulk libraries (EFCore.BulkExtensions) for mass operations.
// N+1 Fix Example
// ❌ N+1 — 1 query for orders + N queries for items
var orders = await _ctx.Orders.ToListAsync();
foreach (var o in orders)
    Console.WriteLine(o.Items.Count); // triggers lazy load

// ✅ Single query with eager load
var orders = await _ctx.Orders
    .AsNoTracking()
    .Include(o => o.Items)
    .ToListAsync();
Senior Signal
Distinguish when to use EF Core vs Dapper vs raw ADO.NET. EF Core for transactional writes; Dapper for complex reads; ADO.NET for bulk operations. Showing this nuance signals data-layer maturity.
06 Security vulnerabilities (e.g., authentication flaws) are identified in your application. How would you address and prevent them? Security
// Immediate Response (War-Room Mode)
1
Assess impact and scope — are credentials or PII exposed? Invoke incident response plan; notify DPO if GDPR-relevant.
2
Patch or mitigate immediately — WAF rule, feature flag, or hot-fix deploy to stop the bleeding.
3
Rotate all potentially-exposed secrets via Azure Key Vault secret rotation.
4
Conduct post-mortem to understand root cause and prevent recurrence.
// ASP.NET Core Security Hardening
  • Authentication: Use Duende IdentityServer or Azure AD B2C with OAuth2 / OIDC. Never hand-roll auth. Enforce short-lived JWTs (15 min) + refresh token rotation.
  • Authorisation: Policy-based with IAuthorizationHandler; resource-based checks. Never rely solely on role strings in JWT claims.
  • Input validation: FluentValidation on every command; model binding is not enough. Validate at the boundary, not in business logic.
  • SQL injection: EF Core parameterises by default; never interpolate user input into raw SQL. Use FromSqlRaw only with parameters.
  • HTTPS enforced: UseHsts(), UseHttpsRedirection(), and HSTS preloading. Minimum TLS 1.2.
  • Security headers: CSP, X-Frame-Options, X-Content-Type-Options via middleware or reverse proxy.
  • Secret management: Zero secrets in code or config files. Azure Key Vault + Managed Identity everywhere.
// Preventive Practices
  • SAST in CI (SonarCloud, Semgrep) — block PRs with high-severity findings.
  • Dependency scanning (Dependabot, OWASP Dependency-Check) — auto-update transitive vulnerabilities.
  • Penetration testing at least annually; threat modelling for each major feature.
  • OWASP Top 10 training mandatory for all engineers, not just security team.
Senior Signal
Frame security as "shift-left" — a development culture, not a gate. Mention threat modelling (STRIDE/DREAD) during design phase and security champions embedded in squads.
07 Your team is facing deployment failures in CI/CD pipelines. How would you stabilize and improve the process? DevOps
// Immediate Stabilisation
1
Categorise failures — flaky tests, infra provisioning errors, artifact issues, or environment drift. Track failure rates per stage in a dashboard.
2
Add mandatory automated rollback — failed health checks after deploy trigger automatic rollback to the previous release (Helm rollback / Azure Deployment Slots swap back).
3
Quarantine flaky tests — move them to a separate suite and fix within the sprint; don't let them block main branch.
// Pipeline Design Principles
  • Fast feedback first: Unit tests → lint → SAST → Docker build → integration tests → staging deploy → smoke tests → production. Fail early, fail cheap.
  • Infrastructure as Code: Bicep / Terraform for all environments. Eliminate manual config drift that causes "works in staging, fails in prod".
  • Feature flags (LaunchDarkly / Azure App Config): Decouple deploy from release. Ship dark, enable gradually via percentage rollout.
  • Progressive delivery: Canary releases (5% → 25% → 100%) with automated promotion/rollback based on error-rate SLO.
  • Immutable artifacts: Build once, promote the same Docker image tag through environments. Never rebuild for each environment.
  • Environment parity: Use Docker Compose or Testcontainers for integration tests — same DB version, same config, no surprises.
Senior Signal
Talk about DORA metrics (deployment frequency, lead time, change failure rate, MTTR) as the north star. Improving CI/CD is not about speed — it's about raising deployment confidence so the team can ship multiple times per day safely.
08 You need to integrate multiple third-party services with varying reliability. How would you design a resilient integration layer? Integration
// Core Resilience Patterns
  • Circuit Breaker (Polly v8): Automatically open the circuit after N consecutive failures; return a fallback or cached response instead of waiting on a dead service.
  • Retry with jitter: Exponential back-off + randomised jitter prevents thundering-herd on recovery. ResiliencePipelineBuilder in Polly 8 composes cleanly.
  • Timeout policy: Every outbound HTTP call must have an explicit timeout. Don't rely on the third-party's timeout.
  • Bulkhead isolation: Separate HttpClient instances and thread budgets per third-party so one slow vendor can't exhaust the global thread pool.
  • Outbox / async messaging: For non-latency-critical calls, publish to a durable queue (Service Bus). A background worker calls the third-party — decoupled, retryable, auditable.
  • Idempotency keys: Include an idempotency key in every mutation request so safe retries don't cause duplicate operations (payments, orders).
// Polly v8 Resilience Pipeline
var pipeline = new ResiliencePipelineBuilder<HttpResponseMessage>()
    .AddRetry(new RetryStrategyOptions<HttpResponseMessage>
    {
        MaxRetryAttempts = 3,
        Delay = TimeSpan.FromSeconds(1),
        BackoffType = DelayBackoffType.Exponential,
        UseJitter = true
    })
    .AddCircuitBreaker(new CircuitBreakerStrategyOptions<HttpResponseMessage>
    {
        FailureRatio = 0.5,
        SamplingDuration = TimeSpan.FromSeconds(30),
        BreakDuration = TimeSpan.FromSeconds(15)
    })
    .AddTimeout(TimeSpan.FromSeconds(5))
    .Build();
// Observability for Third-Party SLAs
  • Track per-vendor latency p95/p99, error rate, and circuit-open frequency as separate metrics.
  • Set up SLA breach alerts — alert when a vendor degrades before your SLO degrades.
  • Maintain a vendor health dashboard so on-call engineers see which integration is the problem immediately.
Senior Signal
Bring up anti-corruption layer (ACL) pattern — wrap each third-party behind an adapter interface so swapping vendors requires no change to domain logic. Also discuss graceful degradation: what does the user see when the integration is unavailable?
09 Technical debt is slowing down development. How would you prioritise and manage refactoring efforts? Engineering Culture
// Classify the Debt
  • Reckless/inadvertent: Bad decisions made unknowingly — highest risk, fix first.
  • Reckless/deliberate: "We'll fix it later" shortcuts — track in backlog with a due date.
  • Prudent/deliberate: Conscious trade-offs ("ship now, refactor after learning") — acceptable with a written plan.
  • Prudent/inadvertent: "Now we know better" — address during natural touch-points.
// Prioritisation Framework
1
Hotspot analysis: Combine code churn (git log) with complexity (NDepend / SonarCloud cognitive complexity). High-churn + high-complexity files are the most painful — fix these first.
2
Business impact score: Rate each debt item by developer velocity loss, defect rate contribution, and risk if unchanged. Quantify in sprint hours lost per quarter.
3
20% rule: Reserve 20% of every sprint for refactoring and test coverage improvement. Frame it as velocity investment to stakeholders, not cleanup.
4
Boy Scout Rule: Every PR must leave the touched code slightly cleaner — rename, add test, extract method. Compound gains over time.
// Making It Visible
  • Maintain a technical debt register in the backlog (tagged, estimated, with business impact) — make debt visible to product, not hidden in engineering.
  • Track tech debt KPIs: code coverage trend, cyclomatic complexity p90, SonarCloud technical debt ratio.
  • Hold quarterly debt retrospectives — celebrate wins, re-prioritise, and keep engineering engaged.
Senior Signal
Articulate the stakeholder conversation: translate technical debt into business risk — "this module has caused 3 P1 incidents costing $X in Q3; a 2-week refactor reduces that risk by 80%." Engineers who speak in business outcomes get budget.
10 The organization wants to migrate on-premise .NET applications to the cloud. How would you plan and execute this transition? Cloud Migration
// The 6 Rs — Choose per Application
Quick Wins
  • Rehost (Lift & Shift) — move VM to Azure VM / Azure Migrate. Minimal risk, minimal gain.
  • Replatform — move .NET Framework to App Service or Azure Container Apps with minimal code change. Sweet spot for most apps.
Long-term Value
  • Refactor/Re-architect — modernise to cloud-native (microservices, serverless). Highest ROI, highest effort.
  • Retire — decommission apps with no active users; discovered during inventory.
// Migration Phases
1
Discover & Assess: Use Azure Migrate to inventory all apps, dependencies, and infra. Classify each app on the 6 Rs. Identify regulatory/data residency constraints.
2
Modernise runtime: Upgrade .NET Framework apps to .NET 8 LTS — this unlocks Linux containers, performance gains, and Azure PaaS compatibility. Use the .NET Upgrade Assistant.
3
Containerise: Docker + Azure Container Registry. Prove the app runs in a container before migrating. Establish base images and vulnerability scanning.
4
Migrate data: SQL Server → Azure SQL (DMS), on-prem Redis → Azure Cache for Redis, file shares → Azure Blob Storage. Run dual-write during cutover.
5
Establish cloud foundations: Landing Zone (hub-spoke VNet), IAM via Azure AD, Key Vault, Azure Policy for governance, cost budgets and alerts.
6
Cutover & decommission: Blue-green cutover with DNS TTL pre-lowered. Run parallel for 2 weeks, then decommission on-prem with backups retained per policy.
// Cost & Governance
  • Right-size VMs using Azure Advisor recommendations; use Reserved Instances for stable workloads (up to 72% savings).
  • Enforce tagging policy (env, team, cost-center) via Azure Policy — essential for chargeback.
  • FinOps review monthly: identify idle resources, oversized SKUs, and egress costs.
Senior Signal
Lead with the people and process side: cloud migration fails more often due to skills gaps and organisational resistance than technical challenges. Propose a Cloud Centre of Excellence (CCoE), an enablement programme, and a Champion network embedded in each team.
